# 我们的 5 阶段全链路测试流程

## 第 1 阶段：需求与逻辑校验

• 完整梳理所有**功能点、规则、流程、边界**
• 核对需求文档与实际实现是否一致
• 找出**逻辑矛盾、规则模糊、未定义行为**
• 列出所有**必须满足的正常流程**与**禁止出现的错误行为**
• 提前预判高风险模块与易出问题环节

## 第 2 阶段：功能全覆盖测试

• 对**每一个按钮、输入、选择、跳转、弹窗**逐一测试
• 正向流程：正常操作是否能跑通
• 反向流程：不按预期操作是否会出问题
• 状态测试：启用/禁用、已完成/未完成、空数据/满数据
• 权限测试：不同角色/身份看到的内容是否正确
• 确保**所有功能点都被覆盖，不遗漏任何一个细节**

## 第 3 阶段：异常与边界场景测试（无死角核心）

• 空值、超长文本、特殊字符、非法输入
• 网络弱网、断网、重连、请求超时
• 重复提交、快速连续点击、并发操作
• 数据极端值：0、负数、极大数、空列表
• 页面刷新、回退、关闭重开、跨页面跳转
• 强制杀死应用/崩溃恢复后的状态一致性
• 找出所有**“用户一乱点就出事”**的场景

## 第 4 阶段：体验、兼容与稳定性测试

• 交互是否流畅：点击反馈、加载状态、错误提示
• 文案是否清晰：无歧义、无错别字、无专业黑话
• 布局与样式：排版错乱、遮挡、溢出、适配问题
• 跨端兼容：移动端/桌面端/不同浏览器/不同屏幕
• 性能：卡顿、延迟、白屏、资源占用异常
• 崩溃、闪退、卡死、数据丢失等严重问题优先标记

## 第 5 阶段：测试报告与回归验收

• 输出结构化Bug清单：

- 问题等级：致命 / 严重 / 一般 / 建议优化
- 复现步骤：1→2→3→4 清晰可复现
- 预期结果 vs 实际结果
- 截图/场景说明
  • 对已修复Bug做**回归测试**，确认彻底解决
  • 给出**上线风险评估**与**必须修复项**
  • 最终出具：可上线 / 建议修复后上线 / 暂不上线

# 测试原则（绝对执行）

## 全覆盖原则

• 不假设“这里应该没问题”
• 不跳过任何一个按钮、输入、流程、状态
• 不遗漏任何异常、边界、极端场景

## 可复现原则

• 所有Bug必须**能稳定复现**
• 步骤清晰，任何人按步骤都能复现问题
• 不接受“偶现但说不清”的模糊描述

## severity 分级原则

- **致命**：导致崩溃、数据丢失、无法使用
- **严重**：功能不可用、逻辑错误、流程阻断
- **一般**：功能可用，但体验不友好
- **优化**：建议提升，但不影响主流程

## 诚实原则

• 有问题直接说，不隐瞒、不美化
• 风险提前讲，不等到上线才暴露
• 不敷衍“能用就行”，追求“绝对可靠”

## 零放过原则

• 哪怕是极小的显示错位、提示错误，也要记录
• 不让任何**隐性问题**带到线上

# 质量红线（绝对不允许上线）

• 崩溃、闪退、卡死
• 数据错误、数据丢失、计算错误
• 流程走不通、关键功能不可用
• 无错误提示、用户不知道错在哪
• 边界场景一触发就异常
• 安全漏洞、越权访问、信息泄露
